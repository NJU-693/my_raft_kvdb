# Raft KV 存储系统 - 架构设计

## 1. 系统概述

Raft KV 存储系统是一个基于 Raft 共识算法的分布式键值存储系统。系统通过 Raft 算法保证多个节点之间的数据一致性，即使在部分节点故障的情况下也能继续提供服务。

### 1.1 设计目标

- **一致性**：保证所有节点的数据最终一致
- **可用性**：在多数节点存活时系统可用
- **容错性**：能够容忍少数节点故障
- **性能**：提供合理的读写性能
- **可理解性**：代码结构清晰，易于学习和维护

## 3. 核心模块

### 3.1 Raft 模块

**职责**：
- 实现 Raft 共识算法
- 管理节点状态转换
- 处理选举和日志复制
- 保证数据一致性

**主要组件**：
- `RaftNode`：Raft 节点主类
- `LogEntry`：日志条目
- `RPC Messages`：RPC 消息定义

### 3.2 存储模块

**职责**：
- 提供高效的 KV 存储
- 实现跳表数据结构
- 管理数据持久化

**主要组件**：
- `SkipList`：跳表实现
- `KVStore`：KV 存储引擎
- `Persister`：持久化管理

### 3.3 网络模块

**职责**：
- 处理节点间通信
- 实现 RPC 服务
- 管理网络连接

**主要组件**：
- `RPCService`：RPC 服务端
- `RPCClient`：RPC 客户端
- 消息序列化/反序列化

### 3.4 工具模块

**职责**：
- 提供通用工具函数
- 日志记录
- 配置管理

**主要组件**：
- `Logger`：日志系统
- `Config`：配置管理
- `ThreadPool`：线程池（可选）

## 4. 数据流

### 4.1 写请求流程

```
1. 客户端发送 Put 请求
   │
   ▼
2. 请求到达某个节点
   │
   ├─ 如果是 Follower ──► 重定向到 Leader
   │
   ▼
3. Leader 接收请求
   │
   ▼
4. Leader 创建日志条目
   │
   ▼
5. Leader 追加到本地日志
   │
   ▼
6. Leader 并行发送 AppendEntries 到所有 Follower
   │
   ▼
7. Follower 检查一致性并追加日志
   │
   ▼
8. Follower 返回成功响应
   │
   ▼
9. Leader 收到多数派响应
   │
   ▼
10. Leader 提交日志（更新 commitIndex）
    │
    ▼
11. Leader 应用到 KV 存储
    │
    ▼
12. Leader 返回结果给客户端
    │
    ▼
13. 下次心跳通知 Follower 提交
```

### 4.2 读请求流程

```
1. 客户端发送 Get 请求
   │
   ▼
2. 请求到达某个节点
   │
   ├─ 如果是 Follower ──► 重定向到 Leader
   │
   ▼
3. Leader 接收请求
   │
   ▼
4. Leader 从 KV 存储读取数据
   │
   ▼
5. Leader 返回结果给客户端
```

### 4.3 选举流程

```
1. Follower 选举超时
   │
   ▼
2. 转变为 Candidate
   │
   ▼
3. 增加 currentTerm
   │
   ▼
4. 投票给自己
   │
   ▼
5. 并行发送 RequestVote 到所有节点
   │
   ▼
6. 其他节点检查并投票
   │
   ▼
7. Candidate 收集投票
   │
   ├─ 获得多数票 ──► 成为 Leader
   │
   ├─ 收到更高任期 ──► 成为 Follower
   │
   └─ 超时 ──► 重新发起选举
```

## 5. 线程模型

### 5.1 主要线程

1. **主线程**：
   - 初始化系统
   - 启动各个模块
   - 处理信号

2. **RPC 服务线程**：
   - 监听网络请求
   - 处理 RPC 调用
   - 多线程处理请求

3. **定时器线程**：
   - 选举超时检查
   - 心跳发送
   - 定期任务

4. **应用线程**：
   - 应用已提交的日志
   - 更新状态机

### 5.2 并发控制

- 使用互斥锁保护共享状态
- 关键路径使用细粒度锁
- 避免死锁和竞态条件

## 6. 持久化策略

### 6.1 需要持久化的数据

1. **Raft 状态**：
   - `currentTerm`：当前任期
   - `votedFor`：投票记录
   - `log[]`：日志条目数组

2. **KV 数据**（可选）：
   - 通过日志重放恢复
   - 或使用快照持久化

### 6.2 持久化时机

- 状态变更时立即持久化
- 使用 fsync 保证数据落盘
- 批量写入优化性能

## 7. 容错机制

### 7.1 节点故障

- **Follower 故障**：Leader 继续服务，故障节点恢复后追赶日志
- **Leader 故障**：触发新一轮选举，选出新 Leader
- **多数节点故障**：系统不可用，等待节点恢复

### 7.2 网络分区

- 少数派分区无法选出 Leader
- 多数派分区继续服务
- 分区恢复后自动同步

### 7.3 数据恢复

- 节点重启后从持久化存储恢复状态
- 通过日志重放恢复 KV 数据
- 自动追赶缺失的日志

## 8. 性能优化

### 8.1 批量操作

- 批量追加日志
- 批量发送 RPC
- 减少网络往返

### 8.2 并行处理

- 并行发送 AppendEntries
- 并行处理客户端请求
- 使用线程池

### 8.3 Pipeline

- 流水线式日志复制
- 不等待响应继续发送
- 提高吞吐量
